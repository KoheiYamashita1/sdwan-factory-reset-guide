Cisco SD-WAN Router Factory Reset Guide

Scope: Cisco SD-WAN routers / Bootstrap file recreation / Factory Reset

Japanese backup: Japanese version

1. Purpose

Securely factory reset a Cisco SD-WAN router with factory reset all secure, restore the OS image, and bring the router back up by using the Bootstrap file bootflash:ciscosdwan.cfg. This is the main procedure. It starts while the router is operating in SD-WAN Controller Mode and covers executing the secure Factory Reset directly from the CLI, recovering the OS image, and re-establishing SD-WAN.

Important: factory reset all secure is irreversible and additionally wipes the bootflash filesystem (including the OS image). Configuration, user information, licenses, OBFL logs, ROMMON variables, user data, and startup configuration are deleted. Always back up the Bootstrap content and required configuration, and prepare a verified OS image on USB or TFTP before proceeding.
Looking for the non-secure reset? If a full secure erase is not required and the OS image can be retained, use the Optional: Basic Reset (factory reset all) page instead.

2. Prerequisites

Console Access Ensure console access is available so the router can be operated after Factory Reset.
Starting Mode Start this procedure while the router is operating in SD-WAN Controller Mode.
Power Do not power off or manually reboot the device during Factory Reset.
Pre-check: Confirm that the computer currently in use is the intended one and that the running router configuration is as expected.

3. Create the Bootstrap Configuration

Run the following command on the router to create the Bootstrap configuration.

request platform software sdwan bootstrap-config save

Checkpoints

  • The command completes without errors.
  • bootflash:ciscosdwan.cfg has been created.

4. Retrieve the Bootstrap Content

Display the generated Bootstrap file and copy the output. On Cisco IOS XE routers, run terminal length 0 so the full content is displayed without stopping. Confirm the display command that is appropriate for your environment. 

terminal length 0
dir bootflash: | include ciscosdwan.cfg
more bootflash:ciscosdwan.cfg
Note: The Bootstrap file contains device-specific information such as UUID, system-ip, site-id, vBond details, hostname, and user configuration. Do not hardcode credentials or secret hashes in the guide. Store them only in an authorized secure location.

5. Prepare the TCLSH Command to Recreate the Bootstrap File

Assuming all files may be deleted after Factory Reset, paste the copied Bootstrap content into the following tclsh command format. At this stage, you are not sending commands to the router. Instead, edit the local procedure document or paste-in configuration file and prepare the command set to be used after Factory Reset. The prepared command will be pasted into the router later through the console connection to recreate bootflash:ciscosdwan.cfg. 

tclsh
puts [open "bootflash:ciscosdwan.cfg" w+] {<Paste the Bootstrap file content here>
}

Model / config-specific note (ISR1100-4G / 6G): In some deployments, an ip helper-address configured under an interface such as interface BDI1 is not reflected in the Bootstrap file or in show sdwan running-config. When this occurs, do not rely on the Bootstrap file. Instead, copy and paste the saved local running configuration (local-config) to recreate the device configuration reliably.

6. Prepare the OS Image for Recovery (USB or TFTP)

Because factory reset all secure wipes the bootflash filesystem, the OS image must be restored after the reset. Before running the reset, prepare an OS image obtained from the official Cisco download page on USB or a reachable TFTP server. Do not reuse the OS image left on the device bootflash. 

Router#show version | include System image
Router#show boot
Router#dir bootflash:
Router#dir usb0:
  1. Confirm the current OS version and the official OS image filename to use for recovery.
  2. Download the matching IOS XE / SD-WAN image from the official Cisco download page to the work computer.
  3. Verify the downloaded file checksum against Cisco's published value.
  4. Place the verified image on USB (root directory) or on a reachable TFTP server.
  5. Confirm that the device can read the image from USB.
Router#dir usb0:
Router#verify /sha512 usb0:<IOSXE_IMAGE.bin>
Replace as needed: The examples use <IOSXE_IMAGE.bin> as the OS image filename and usb0: as the USB device name. The USB device name may be usbflash1: or another name depending on the model and slot. Adjust to match the actual device.
Required: Do not start the secure Factory Reset until the OS image checksum is verified and the device can read it from USB or TFTP.

7. Execute factory reset all secure

While the router is operating in SD-WAN Controller Mode, execute the secure Factory Reset with the following command. Review the deletion scope and impact at each confirmation prompt before continuing. 

Router#factory reset all secure

Several confirmation prompts are displayed. Review the content and press Enter to continue.

The factory reset operation is irreversible for all operations. Are you sure? [confirm]

The following will be deleted as a part of factory reset:
 1: All writable file systems and personal data
 2: OBFL logs
 3: Licenses
 4: Userdata and Startup config
 5: Rommon variables
 6: User Credentials

The system will reload to perform factory reset.
This operation can take anywhere between 30 minutes to 3 hours
DO NOT UNPLUG THE POWER OR INTERRUPT THE OPERATION
Are you sure you want to continue? [confirm]
Do not: During Factory Reset, do not power off the device, remove the USB media, force reboot, or interrupt the operation from the console. Completion can take approximately 30 minutes to 3 hours.
ISR1100-4G / 6G note: With factory reset all secure (3-pass / 7-pass), the bootflash filesystem itself is wiped and can no longer be referenced, so booting from TFTP or USB is mandatory (verified with TFTP in a lab). In ROMMON you may need to temporarily set the console baud rate to 115200; the configuration method is otherwise the same.

8. Boot from USB/TFTP and Restore the OS Image

After the secure reset, no bootable OS image exists on bootflash, so the router boots into ROMMON. Boot temporarily from the OS image on USB (or TFTP), respond to the initial prompts (see step 9) to reach the IOS command prompt, then copy the image back to bootflash and set the BOOT variable. 

rommon > dir usb0:
rommon > boot usb0:<IOSXE_IMAGE.bin>
ISR1100-4G / 6G note: On the ISR1100-4G / 6G, you may need to temporarily change the console baud rate to 115200 before the ROMMON prompt is usable. Restore the original baud rate after the router boots. If the USB device name differs, check with dir usbflash1:. TFTP boot (for example boot tftp://<TFTP_SERVER>/<IOSXE_IMAGE.bin>) can be used instead of USB.

Copy the OS Image to bootflash

Router#dir usb0:
Router#dir bootflash:
Router#copy usb0:<IOSXE_IMAGE.bin> bootflash:<IOSXE_IMAGE.bin>
Router#verify /sha512 bootflash:<IOSXE_IMAGE.bin>

Set the BOOT Variable and Verify

Router#configure terminal
Router(config)#no boot system
Router(config)#boot system bootflash:<IOSXE_IMAGE.bin>
Router(config)#config-register 0x2102
Router(config)#end
Router#write memory
Router#show boot
  • The System image in show version points to bootflash:<IOSXE_IMAGE.bin>.
  • show boot shows that the BOOT variable points to the OS image on bootflash.
  • The router boots successfully even with USB removed.
Note: Do not save the SD-WAN configuration to NVRAM yet. Continue to recreate the Bootstrap file and enable Controller Mode in the next steps.

9. Respond to Initial Prompts After Factory Reset

After executing Factory Reset, wait several hours for completion. When the following System Configuration Dialog appears, do not enter the initial configuration dialog. Enter no. 

         --- System Configuration Dialog ---

Would you like to enter the initial configuration dialog? [yes/no]: no

You will then be prompted for an enable secret. Enter a temporary password. This password is only used to continue the recovery steps that follow. 

Enter enable secret: <Enter a temporary password>
Confirm enable secret: <Re-enter the temporary password>

After entering the password, mode transition options are displayed. Select 0 to proceed to the IOS command prompt without saving the configuration. 

[0] Go to the IOS command prompt without saving this config.
[1] Return back to the setup without saving this config.
[2] Save this configuration to nvram and exit.

Enter your selection [2]: 0

Confirm Autonomous Mode

After reaching the IOS command prompt, run show version and confirm that Autonomous appears in the output. This confirms that the router booted in Autonomous Mode after Factory Reset. 

Router#show version
<Confirm that Autonomous appears in the output>
Note: Do not save the configuration to NVRAM at this stage. Recreate the Bootstrap file first, then perform the required boot and application checks in later steps.

11. Enable Controller Mode

After recreating and verifying the Bootstrap file, enable Controller Mode with the following command. 

Router#controller-mode enable

Enabling Controller Mode erases the NVRAM filesystem and configuration files, then reloads the device. Review the confirmation prompts and press Enter. 

Enabling controller mode will erase the nvram filesystem, remove all configuration files, and reload the box!
Ensure the BOOT variable points to a valid image
Continue? [confirm]
WARNING:
Boot variable either does not exist or buffer is too small
This may impact autoboot of the router. Proceed with caution
Do you wish to proceed with reload anyway[confirm]

Example Log After Execution

*May 20 04:24:20.549: %IOSXE_RP_CFG_NOT-3-BOOT_VAR_NOT_SET: Either the boot variable does not exist or the buffer is too small.Keep it blank.
*May 20 04:24:21.177: %SYS-7-NV_BLOCK_INIT: Initialized the geometry of nvram
*May 20 04:24:21.725: %BOOT-6-MODECHANGE: Mode change success
Note: After controller-mode enable, configuration files are deleted and the device reloads. A BOOT variable warning may be displayed, so verify the boot image state before executing this command.

12. Verify SD-WAN Health After Reboot

After the reboot following Controller Mode enablement completes, confirm that the SD-WAN configuration was loaded from the Bootstrap file and that the router can connect to the SD-WAN controllers.

Check Control Connections

Check connectivity to SD-WAN controllers such as vBond, vSmart, and vManage. Confirm that the expected controller connections are established and healthy. 

show sdwan control-connections

Check BFD Sessions

Check BFD session state over SD-WAN tunnels. Confirm that the required TLOC / peer sessions are established. 

show sdwan bfd sessions

Check the Global Routing Table

Confirm that required routes exist in the global routing table. 

show ip route

Check VRF Routing Tables

Check routes on the service VPN / VRF side. Replace XX with the target VRF number. 

show ip route vrf <XX>

Verification Points

  • show sdwan control-connections shows established SD-WAN controller connections.
  • show sdwan bfd sessions shows the required BFD sessions.
  • show ip route shows the required management or underlay routes.
  • show ip route vrf <XX> shows the required service VPN / VRF routes.

Procedure When the Bootstrap File Is Not Loaded Correctly

Scope: Cisco SD-WAN routers using Firewall features, Cisco ThousandEyes, or similar advanced features

Purpose of this page: When using advanced features such as Firewall or Cisco ThousandEyes, configuration may not be loaded correctly when booting with the Bootstrap file (bootflash:ciscosdwan.cfg) from the main procedure. Use this workaround procedure for those deployments.

Difference from the Main Procedure

This procedure does not rely on loading configuration from the Bootstrap file (bootflash:ciscosdwan.cfg). Instead, perform CLI configuration in Controller Mode and manually enter the minimum required settings (system parameters, VPN 0 transport, and tunnel interface) to establish control connections.

The minimum configuration approach follows Cisco's official SD-WAN WAN Edge Onboarding Deploy Guide section "Option 3: Manual deployment for IOS-XE SD-WAN device."

Starting point: This procedure starts from the state where Controller Mode has already been enabled (controller-mode enable) and the router has booted in Controller Mode without correctly loading the Bootstrap configuration. Factory Reset execution, initial prompt handling, and Controller Mode enablement are assumed to have already been completed according to the main procedure.
Workflow:
  1. Stop the PnP process and manually enter minimum configuration.
  2. Check Control Connections

A-1. Stop PnP and Enter Minimum Configuration

After booting in Controller Mode, the router automatically starts the PnP process. Stop the PnP process first so that configuration can be entered manually. 

pnpa service discovery stop
Note: Immediately after Factory Reset, the default SD-WAN WAN Edge credentials are admin/admin. To save configuration changes, run commit in configuration mode.

Step 1: Configure system parameters

Configure hostname, system-ip, site-id, organization-name, and vBond (IP address or FQDN). Also specify DNS servers with ip name-server.

Required: If vBond is specified as an FQDN, the router cannot connect to controllers unless DNS resolution works. Configure DNS servers with ip name-server <DNS_SERVER_IP>. This is not required if DNS servers are automatically assigned by DHCP or a similar mechanism. 
config-transaction
system
 system-ip            <SYSTEM_IP>
 site-id              <SITE_ID>
 organization-name    <ORGANIZATION_NAME>
 vbond                <VBOND_IP_OR_FQDN>
 exit
!
hostname <HOSTNAME>
ip name-server <DNS_SERVER_IP>
exit
Uncommitted changes found, commit them? [yes/no/CANCEL] yes
Commit complete.

Step 2: Configure the transport (VPN 0) WAN interface and tunnel

Configure transport VPN 0 for controller reachability. Assign an IP address to the WAN interface, then configure encapsulation, color, and routing (dynamic or default route) on the tunnel interface. WAN Edge establishes secure DTLS/TLS control-plane connections only after the tunnel interface is configured. 

config-transaction
interface <WAN_INTERFACE>
 description <TRANSPORT_DESCRIPTION>
 no shutdown
 ip address <WAN_IP> <WAN_MASK>
 exit
!
interface Tunnel<TUNNEL_ID>
 no shutdown
 ip unnumbered <WAN_INTERFACE>
 ipv6 unnumbered <WAN_INTERFACE>
 tunnel source <WAN_INTERFACE>
 tunnel mode sdwan
 exit
!
sdwan
 interface <WAN_INTERFACE>
  tunnel-interface
   encapsulation ipsec
   color <COLOR>
  exit
 exit
!
ip route 0.0.0.0 0.0.0.0 <WAN_GATEWAY>
end
Uncommitted changes found, commit them? [yes/no/CANCEL] yes
Commit complete.
Note: Replace <COLOR> (for example, mpls or biz-internet), interface names, and routes according to the actual device configuration. If multiple transports are used, configure each WAN interface similarly.

A-2. Verify Control Connections and Health

After entering the minimum configuration, confirm that connections to SD-WAN controllers (vBond / vSmart / vManage) are established. After control connections are established, if a device template is assigned in vManage, configuration is downloaded and overwrites the current configuration.

Check Control Connections

show sdwan control-connections

Check BFD Sessions

show sdwan bfd sessions

Check Routing Tables

show ip route
show ip route vrf <XX>

Verification Points

  • show sdwan control-connections shows established SD-WAN controller connections.
  • show sdwan bfd sessions shows the required BFD sessions.
  • Required routes are visible in show ip route / show ip route vrf <XX>.
  • The target device reachability is healthy in vManage and template attachment has completed.
Recovering Firewall / ThousandEyes features:

Even after control connections are established and a device template is applied from vManage, Firewall and ThousandEyes features are not restored by that alone. These features require additional vManage-side procedures separate from the minimum configuration entered in this procedure.

  • Firewall (UTD) features: Upload the Snort-based UTD (Unified Threat Defense) engine as a Security Virtual Image to the vManage software repository, then attach the required security policy / container profile to the device template and install it.
  • ThousandEyes features: Upload the ThousandEyes Enterprise Agent image to vManage, create and attach the ThousandEyes Agent feature template (including Account Group Token, etc.), and install the agent.

The detailed procedures for these features are outside the scope of this guide. Refer to the official documentation below.

  • Firewall (UTD): Cisco Catalyst SD-WAN Security Configuration Guide - Security Virtual Image / Install UTD Security Virtual Image on cEdge Routers
  • ThousandEyes: Configure ThousandEyes on SD-WAN Devices / Installing Enterprise Agents on Cisco Routers using SD-WAN Manager Feature Templates

Optional: Basic Reset (factory reset all)

Scope: Use when a full secure erase is not required and the OS image can be retained.

When to use this option: The main procedure uses factory reset all secure, which wipes the bootflash filesystem including the OS image. When a full secure erase is not required (for example, device reuse or relocation), you can use factory reset all instead. In that case packages.conf and the OS image remain, so the USB/TFTP OS image recovery steps are not needed.
Important: factory reset all is still irreversible for configuration, user data, licenses, OBFL logs, ROMMON variables, and startup config. Back up the Bootstrap content and required configuration before proceeding.

Difference from the Main Procedure

The preparation (Bootstrap creation and TCLSH preparation) and the post-reset steps (initial prompts, Bootstrap recreation, Controller Mode enablement, and SD-WAN verification) are identical to the main procedure. Only the reset command and the OS image handling differ.

  • Reset command: factory reset all (instead of factory reset all secure).
  • OS image: retained on bootflash, so USB/TFTP OS recovery is not required.
  • Boot after reset: follows the config-register value (does not always fall back to ROMMON).

Execute factory reset all

While the router is operating in SD-WAN Controller Mode, execute Factory Reset directly with the following command. 

factory reset all

Several confirmation prompts are displayed. Review the content and press Enter to continue.

The factory reset operation is irreversible for all operations. Are you sure? [confirm]

The following will be deleted as a part of factory reset:
 1: All writable file systems and personal data
 2: OBFL logs
 3: Licenses
 4: Userdata and Startup config
 5: Rommon variables
 6: User Credentials

The system will reload to perform factory reset.
This operation can take anywhere between 30 minutes to 3 hours
DO NOT UNPLUG THE POWER OR INTERRUPT THE OPERATION
Are you sure you want to continue? [confirm]

Factory reset will take a backup of the boot image if the system is currently booted from an image stored locally.
If the current boot image is a remote image or stored on a usb/nim-ssd, please take a backup of the image before executing this command.
Are you sure you want to continue? [confirm]
Do not: Factory Reset can take approximately 30 minutes to 3 hours. Do not unplug power, force reboot, or interrupt the operation until the Factory Reset operation and reboot have fully completed.

Optional: If the Router Boots into ROMMON Mode

Note: If the router boots into ROMMON mode after Factory Reset, boot it again with the following commands. Whether this is required depends on the chassis model. 
rommon > confreg 0x2102
rommon > boot
ISR1100-4G / 6G note: On the ISR1100-4G / 6G, in ROMMON you may need to temporarily change the console baud rate to 115200. Restore the original baud rate after the router boots.

Continue with the Main Procedure

After the reset completes, the remaining steps are the same as the main procedure: respond to the initial prompts, recreate the Bootstrap file, enable Controller Mode, and verify SD-WAN health.

C-2. Reset Behavior and Operational Comparison

C-3. Which Command to Use

factory reset all secure Use this when you need complete erasure. Suitable for RMA return, disposal, compromised devices, or securely removing sensitive data. The boot image is also erased, so prepare an official image beforehand.
factory reset all Standard reset. Suitable when reusing the device, moving it to another site, and retaining the configuration register and local boot image.
factory reset keep-licensing-info Use this when resetting the device while retaining license information. Suitable when you want to avoid re-registration or SLAC retrieval work.
factory reset sed Use this when erasing only SED drives. Applies to C8550-G2 / C8570-G2 when bootflash is SED-enabled. Hard disks are not included.

C-4. Common Pre-execution Notes

Required: Always back up images, configuration, and personal data before execution. Do not power off the device during reset. Do not execute while ISSU / ISSD is in progress. Applied software patches are not restored after reset. If ZTP is configured, the device restarts with the ZTP configuration after reset completes.
