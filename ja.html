Cisco SD-WAN ルータ Factory Reset 手順書

対象: Cisco SD-WAN ルータ / Bootstrap ファイル再作成 / Factory Reset

English: Main page

1. 目的

Cisco SD-WAN ルータを factory reset all secure でセキュアに Factory Reset し、OS イメージを復旧したうえで、 リセット後に Bootstrap ファイル bootflash:ciscosdwan.cfg から再度ルータを立ち上げ直す。 本手順はメイン手順であり、ルータが SD-WAN の Controller Mode で動作している状態から開始し、 CLI でのセキュア Factory Reset 実行、OS イメージ復旧、SD-WAN 再確立までを対象とする。

重要: factory reset all secure は不可逆操作であり、加えて bootflash のファイルシステム（OS イメージを含む）も消去される。 設定、ユーザ情報、ライセンス、OBFLログ、ROMMON変数、ユーザデータ、起動設定などが削除される。 実行前に必ず Bootstrap 内容と必要な設定情報を退避し、検証済みの OS イメージを USB または TFTP に準備しておくこと。
セキュアでない通常のリセットを行う場合: 完全消去が不要で OS イメージを残せる場合は、 オプション: 基本リセット（factory reset all）ページを使用する。

2. 前提条件

コンソール接続 Factory Reset 後に操作できるよう、コンソール接続を確保しておく。
開始時の動作モード SD-WAN の Controller Mode で動作している状態から作業を開始する。
電源 Factory Reset 中は電源断や再起動操作を行わない。
作業開始前の確認: 現在動作中のコンピューターが想定通りのものであり、現在動作中のルータの Config が想定通りであることを確認してください。

3. Bootstrap 設定の作成

ルータ上で以下のコマンドを実行し、Bootstrap 設定を作成する。

request platform software sdwan bootstrap-config save

確認ポイント

  • コマンドがエラーなく完了すること。
  • bootflash:ciscosdwan.cfg が作成されていること。

4. Bootstrap 内容の取得

作成された Bootstrap ファイルの内容を表示し、出力内容をコピーする。 Cisco IOS XE ルーターでは terminal length 0 を実行し、途中で停止させずに全体を表示する。 表示コマンドは環境に合わせて確認すること。 

terminal length 0
dir bootflash: | include ciscosdwan.cfg
more bootflash:ciscosdwan.cfg
注意: Bootstrap 内には UUID、system-ip、site-id、vBond 情報、ホスト名、ユーザ設定などの機器固有情報が含まれる。 認証情報や secret hash は手順書に実値を固定せず、権限のある保管場所で管理すること。

5. TCLSH 用 Bootstrap 再作成コマンドの準備

Factory Reset 後に全ファイルが削除された場合を想定し、コピーした Bootstrap 内容を以下の tclsh コマンド形式に貼り付ける。 この手順では、ルータへコマンドを送信するのではなく、手元の作業手順書または 流し込み用コンフィグを編集し、Factory Reset 後に使用する投入用コマンドを作成する。 作成したコマンドは、後続手順でコンソール接続からルータへ流し込み、 bootflash:ciscosdwan.cfg を再作成するために使用する。 

tclsh
puts [open "bootflash:ciscosdwan.cfg" w+] {<ここに Bootstrap ファイルの内容を貼り付ける>
}

貼り付けフォーマット例

tclsh
puts [open "bootflash:ciscosdwan.cfg" w+] {Content-Type: multipart/mixed; boundary="===============<BOUNDARY_ID>=="
MIME-Version: 1.0

--===============<BOUNDARY_ID>==
Content-Type: text/cloud-config; charset="us-ascii"
MIME-Version: 1.0
Content-Transfer-Encoding: 7bit
Content-Disposition: attachment; filename="cloud-config"

#cloud-config
vinitparam:
 - uuid : <DEVICE_UUID>

--===============<BOUNDARY_ID>==
Content-Type: text/cloud-boothook; charset="us-ascii"
MIME-Version: 1.0
Content-Transfer-Encoding: 7bit
Content-Disposition: attachment; filename="config-<DEVICE_UUID>.txt"

#cloud-boothook
system
 system-ip             <SYSTEM_IP>
 site-id               <SITE_ID>
 organization-name     <ORGANIZATION_NAME>
 vbond                 <VBOND_FQDN> port <VBOND_PORT>
!
hostname <HOSTNAME>
username admin privilege 15 secret 9 <ADMIN_SECRET_HASH>
!
<以降、実機から取得した Bootstrap 設定を貼り付ける>
--===============<BOUNDARY_ID>==--
}
機種・構成固有の注意（ISR1100-4G / 6G）: 一部の構成では、interface BDI1 などのインターフェース配下に設定した ip helper-address が、 Bootstrap ファイルや show sdwan running-config に反映されない場合がある。 この場合は Bootstrap ファイルに頼らず、保存しておいたローカルの running-config（local-config）を コピー＆ペーストで投入する方が確実である。

6. 復旧用 OS イメージの準備（USB または TFTP）

factory reset all secure は bootflash のファイルシステムごと消去するため、リセット後に OS イメージを復旧する必要がある。 リセット実行前に、Cisco 公式ダウンロードページから取得した OS イメージを USB または到達可能な TFTP サーバーへ準備しておく。 機器の bootflash に残っている OS イメージは流用しない。 

Router#show version | include System image
Router#show boot
Router#dir bootflash:
Router#dir usb0:
  1. 現在の OS バージョンと、復旧に使用する公式 OS イメージファイル名を確認する。
  2. 対象機種に対応する IOS XE / SD-WAN イメージを Cisco 公式ダウンロードページから作業端末へダウンロードする。
  3. ダウンロードしたファイルのチェックサムを、Cisco 掲載値と一致するか確認する。
  4. 確認済みイメージを USB のルートディレクトリ、または到達可能な TFTP サーバーへ配置する。
  5. 機器から USB 上のイメージが読み出せることを確認する。
Router#dir usb0:
Router#verify /sha512 usb0:<IOSXE_IMAGE.bin>
置き換え: 例では OS イメージファイル名を <IOSXE_IMAGE.bin>、USB のデバイス名を usb0: として記載する。 USB のデバイス名は機種や挿入スロットにより usbflash1: などになる場合がある。実機に合わせて読み替えること。
必須: OS イメージのチェックサム確認と、USB または TFTP から読み出せることの確認が完了するまで、セキュア Factory Reset を開始しないこと。

7. factory reset all secure の実行

SD-WAN の Controller Mode で動作している状態から、以下のコマンドでセキュア Factory Reset を実行する。 各確認プロンプトで削除対象と影響を確認したうえで続行する。 

Router#factory reset all secure

確認プロンプトが複数回表示されるため、内容を確認して Enter を押す。

The factory reset operation is irreversible for all operations. Are you sure? [confirm]

The following will be deleted as a part of factory reset:
 1: All writable file systems and personal data
 2: OBFL logs
 3: Licenses
 4: Userdata and Startup config
 5: Rommon variables
 6: User Credentials

The system will reload to perform factory reset.
This operation can take anywhere between 30 minutes to 3 hours
DO NOT UNPLUG THE POWER OR INTERRUPT THE OPERATION
Are you sure you want to continue? [confirm]
禁止事項: Factory Reset 中は電源断、USB 抜去、強制再起動、コンソール入力による割り込みを行わないこと。 完了まで 30分から3時間程度かかる場合がある。
ISR1100-4G / 6G の注意: factory reset all secure（3-pass / 7-pass）では bootflash がファイルシステムごと消去され参照できない状態になるため、 TFTP または USB からの起動が必須となる（ラボでは TFTP で検証）。ROMMON 時にコンソールのボーレートを一時的に 115200 へ変更する必要がある場合がある。設定方法はその他は同じである。

8. USB/TFTP から起動して OS イメージを復旧

セキュアリセット後は bootflash に起動可能な OS イメージが存在しないため、ルータは ROMMON で起動する。 USB（または TFTP）上の OS イメージから一時的に起動し、初期プロンプト対応（手順 9 参照）で IOS コマンドプロンプトへ進んだうえで、 OS イメージを bootflash へコピーし直し、BOOT 変数を設定する。 

rommon > dir usb0:
rommon > boot usb0:<IOSXE_IMAGE.bin>
ISR1100-4G / 6G の注意: ISR1100-4G / 6G では、ROMMON プロンプトを利用できるようにするため、コンソールのボーレートを一時的に 115200 へ変更する必要がある場合がある。 起動後は元のボーレートへ戻すこと。USB のデバイス名が異なる場合は dir usbflash1: などで確認する。 USB の代わりに TFTP 起動（例: boot tftp://<TFTP_SERVER>/<IOSXE_IMAGE.bin>）も使用できる。

OS イメージを bootflash へコピー

Router#dir usb0:
Router#dir bootflash:
Router#copy usb0:<IOSXE_IMAGE.bin> bootflash:<IOSXE_IMAGE.bin>
Router#verify /sha512 bootflash:<IOSXE_IMAGE.bin>

BOOT 変数の設定と確認

Router#configure terminal
Router(config)#no boot system
Router(config)#boot system bootflash:<IOSXE_IMAGE.bin>
Router(config)#config-register 0x2102
Router(config)#end
Router#write memory
Router#show boot
  • show version の System image が bootflash:<IOSXE_IMAGE.bin> を指していること。
  • show boot で BOOT 変数が bootflash 上の OS イメージを指していること。
  • USB を抜いた状態でも正常に起動できること。
補足: この段階では SD-WAN 設定を NVRAM へ保存しない。続く手順で Bootstrap ファイルの再作成と Controller Mode の有効化を行う。

9. Factory Reset 後の初期プロンプト対応

Factory Reset 実行後、完了まで数時間待機する。以下の System Configuration Dialog が表示されたら、初期設定ダイアログには入らず no を入力する。 

         --- System Configuration Dialog ---

Would you like to enter the initial configuration dialog? [yes/no]: no

続けて enable secret の入力を求められるため、一時的なパスワードを入力する。 ここで入力するパスワードは、この後の作業を進めるための一時的なものとする。 

Enter enable secret: <一時パスワードを入力>
Confirm enable secret: <一時パスワードを再入力>

パスワード入力後、動作モード移行時の選択肢が表示される。 ここでは設定を保存せず IOS コマンドプロンプトへ進むため、0 を選択する。 

[0] Go to the IOS command prompt without saving this config.
[1] Return back to the setup without saving this config.
[2] Save this configuration to nvram and exit.

Enter your selection [2]: 0

Autonomous Mode の確認

IOS コマンドプロンプトへ移行したら、show version を実行し、 出力内に Autonomous の文字列が表示されていることを確認する。 これにより、Factory Reset 後の起動状態が Autonomous Mode であることを確認する。 

Router#show version
<出力内に Autonomous が表示されていることを確認>
補足: この段階では NVRAM へ設定を保存しない。Bootstrap ファイルを再作成してから、 後続手順で必要な起動・適用確認を行う。

Router#tclsh
Router(tcl)#puts [open "bootflash:ciscosdwan.cfg" w+] {Content-Type: multipart/mixed; boundary="===============<BOUNDARY_ID>=="
MIME-Version: 1.0

MIME-Version: 1.0

MIME-Version: 1.0

補足: 実機では複数行貼り付け中に +> の継続プロンプトが表示される場合がある。 +> はルータ側の表示であり、入力する文字ではない。
注意: Bootstrap 内容は必ず { と } の中に貼り付ける。 secret hash などに含まれる $ 文字が Tcl で変換されないようにするため、 ダブルクォートでは囲まないこと。

11. Controller Mode の有効化

Bootstrap ファイルを再作成し、内容確認が完了したら、 以下のコマンドで Controller Mode を有効化する。 

Router#controller-mode enable

Controller Mode の有効化により NVRAM ファイルシステムと設定ファイルが削除され、 機器が再起動される。表示される確認プロンプトの内容を確認し、Enter を押す。 

Enabling controller mode will erase the nvram filesystem, remove all configuration files, and reload the box!
Ensure the BOOT variable points to a valid image
Continue? [confirm]
WARNING:
Boot variable either does not exist or buffer is too small
This may impact autoboot of the router. Proceed with caution
Do you wish to proceed with reload anyway[confirm]

実行後ログ例

*May 20 04:24:20.549: %IOSXE_RP_CFG_NOT-3-BOOT_VAR_NOT_SET: Either the boot variable does not exist or the buffer is too small.Keep it blank.
*May 20 04:24:21.177: %SYS-7-NV_BLOCK_INIT: Initialized the geometry of nvram
*May 20 04:24:21.725: %BOOT-6-MODECHANGE: Mode change success
注意: controller-mode enable 実行後は設定ファイル削除と再起動が行われる。 BOOT 変数に関する警告が表示される場合があるため、事前に起動イメージの状態を確認してから実行すること。

12. 再起動後の SD-WAN 正常性確認

Controller Mode 有効化後の再起動が完了したら、Bootstrap から SD-WAN Config が正常に読み込まれ、 SD-WAN コントローラと接続できていることを確認する。

コントロール接続の確認

vBond、vSmart、vManage などの SD-WAN コントローラとの接続状態を確認する。 想定するコントローラへの接続が確立していること、接続状態が正常であることを確認する。 

show sdwan control-connections

BFD セッションの確認

SD-WAN トンネル上の BFD セッション状態を確認する。 必要な TLOC / peer とのセッションが確立していることを確認する。 

show sdwan bfd sessions

グローバルルーティングテーブルの確認

グローバルルーティングテーブルに必要な経路が存在することを確認する。 

show ip route

VRF ルーティングテーブルの確認

サービス VPN / VRF 側の経路を確認する。XX は確認対象の VRF 番号に置き換える。 

show ip route vrf <XX>

確認観点

  • show sdwan control-connections で SD-WAN コントローラとの接続が確立していること。
  • show sdwan bfd sessions で必要な BFD セッションが確立していること。
  • show ip route で管理系またはアンダーレイ側の必要経路が確認できること。
  • show ip route vrf <XX> でサービス VPN / VRF 側の必要経路が確認できること。

Bootstrap ファイルが正しく読み込まれずに立ち上がってしまった場合の手順

対象: Firewall 機能 / Cisco ThousandEyes などを使用している Cisco SD-WAN ルータ

このページの位置づけ: Firewall 機能や Cisco ThousandEyes などの高度な機能を使用している場合、 メイン手順の Bootstrap ファイル（bootflash:ciscosdwan.cfg）による起動では設定が正しく読み込まれないことが確認されている。 そのため、こうした構成では本ページの回避手順を使用する。

メイン手順との違い

本手順では、Bootstrap ファイル（bootflash:ciscosdwan.cfg）からの設定読み込みに頼らず、 Controller Mode 上で CLI による設定操作を行い、 コントローラと接続するために必要最低限の設定（system パラメータ、VPN 0 トランスポート、Tunnel インターフェース） を手動投入してコントロール接続を確立する。

最低限の設定の考え方は、Cisco 公式の SD-WAN WAN Edge Onboarding Deploy Guide の「Option 3: Manual deployment for IOS-XE SD-WAN device」に準拠する。

本手順の開始地点: 本手順は、すでに Controller Mode の有効化（controller-mode enable）が完了し、 Bootstrap が正しく読み込まれないまま Controller Mode で立ち上がってきた状態からスタートする。 Factory Reset の実行、初期プロンプト対応、Controller Mode の有効化まではメイン手順のとおり実施済みであることを前提とする。
作業の全体像:
  1. PnP プロセスの停止と最低限の設定の手動投入
  2. コントロール接続の確認

A-1. PnP プロセスの停止と最低限の設定投入

Controller Mode 起動後、ルータは起動時に PnP プロセスを自動的に開始する。 手動で設定を投入するため、まず PnP プロセスを停止する。 

pnpa service discovery stop
補足: Factory Reset 直後の SD-WAN WAN Edge のデフォルト認証情報は admin/admin である。 設定変更を保存するには、コンフィグレーションモードで commit を実行する。

Step 1: system パラメータの設定

hostname、system-ip、site-id、organization-name、vBond（IP アドレスまたは FQDN）を設定する。 あわせて ip name-server で DNS サーバーを指定する。

必須: vBond を FQDN で指定している場合、名前解決ができないとコントローラへ接続できない。 ip name-server <DNS_SERVER_IP> による DNS サーバーの設定を必ず行うこと。 ただし、DHCP などの環境で DNS サーバーが自動的に設定される場合は、この限りではない。 
config-transaction
system
 system-ip            <SYSTEM_IP>
 site-id              <SITE_ID>
 organization-name    <ORGANIZATION_NAME>
 vbond                <VBOND_IP_OR_FQDN>
 exit
!
hostname <HOSTNAME>
ip name-server <DNS_SERVER_IP>
exit
Uncommitted changes found, commit them? [yes/no/CANCEL] yes
Commit complete.

Step 2: トランスポート（VPN 0）WAN インターフェースと Tunnel の設定

コントローラへ到達するためのトランスポート VPN 0 を設定する。WAN インターフェースに IP アドレスを設定し、 Tunnel インターフェースに encapsulation・color・ルーティング（動的またはデフォルトルート）を設定する。 Tunnel インターフェースを設定して初めて、WAN Edge は DTLS/TLS によるセキュアなコントロールプレーン接続を確立する。 

config-transaction
interface <WAN_INTERFACE>
 description <TRANSPORT_DESCRIPTION>
 no shutdown
 ip address <WAN_IP> <WAN_MASK>
 exit
!
interface Tunnel<TUNNEL_ID>
 no shutdown
 ip unnumbered <WAN_INTERFACE>
 ipv6 unnumbered <WAN_INTERFACE>
 tunnel source <WAN_INTERFACE>
 tunnel mode sdwan
 exit
!
sdwan
 interface <WAN_INTERFACE>
  tunnel-interface
   encapsulation ipsec
   color <COLOR>
  exit
 exit
!
ip route 0.0.0.0 0.0.0.0 <WAN_GATEWAY>
end
Uncommitted changes found, commit them? [yes/no/CANCEL] yes
Commit complete.
補足: <COLOR>（例: mpls、biz-internet など）やインターフェース名・経路は、 対象機器の実構成に合わせて置き換える。複数トランスポートを使用する場合は、各 WAN インターフェースに対して同様に設定する。

A-2. コントロール接続と正常性の確認

最低限の設定投入後、SD-WAN コントローラ（vBond / vSmart / vManage）との接続が確立されることを確認する。 コントロール接続確立後、vManage にデバイステンプレートが割り当てられている場合は、 設定がダウンロードされ既存設定が上書きされる。

コントロール接続の確認

show sdwan control-connections

BFD セッションの確認

show sdwan bfd sessions

ルーティングテーブルの確認

show ip route
show ip route vrf <XX>

確認観点

  • show sdwan control-connections で SD-WAN コントローラとの接続が確立していること。
  • show sdwan bfd sessions で必要な BFD セッションが確立していること。
  • show ip route / show ip route vrf <XX> で必要経路が確認できること。
  • vManage 上で対象機器の Reachability が正常であり、テンプレート適用が完了していること。
Firewall / ThousandEyes 機能の復旧について:

コントロール接続が確立し、vManage からデバイステンプレートが適用されても、 Firewall や ThousandEyes の機能はそれだけでは復旧しない。 これらの機能は、本手順で投入する最低限の設定とは別に、vManage 側で追加の手順が必要となる。

  • Firewall（UTD）機能: Snort ベースの UTD（Unified Threat Defense）エンジンを Security Virtual Image として vManage のソフトウェアリポジトリへアップロードし、セキュリティポリシー／コンテナプロファイルを デバイステンプレートへ付与してインストールする必要がある。
  • ThousandEyes 機能: ThousandEyes Enterprise Agent のイメージを vManage へアップロードし、 ThousandEyes Agent フィーチャーテンプレート（Account Group Token 等）を作成・付与して エージェントを追加インストールする必要がある。

これらの具体的な手順は本手順書の対象外とする。詳細は以下の公式ドキュメントを参照すること。

  • Firewall（UTD）: Cisco Catalyst SD-WAN Security Configuration Guide - Security Virtual Image ／ Install UTD Security Virtual Image on cEdge Routers
  • ThousandEyes: Configure ThousandEyes on SD-WAN Devices ／ Installing Enterprise Agents on Cisco Routers using SD-WAN Manager Feature Templates

オプション: 基本リセット（factory reset all）

対象: 完全消去が不要で、OS イメージを残せる場合に使用する。

このオプションを使う場面: メイン手順では factory reset all secure を使用し、bootflash のファイルシステム（OS イメージを含む）ごと消去する。 完全消去が不要な場合（例: 機器の再利用や移設）は、代わりに factory reset all を使用できる。 この場合 packages.conf や OS イメージは残るため、USB/TFTP による OS イメージ復旧手順は不要である。
重要: factory reset all でも、設定・ユーザデータ・ライセンス・OBFLログ・ROMMON変数・起動設定の消去は不可逆である。 実行前に Bootstrap 内容と必要な設定情報を退避すること。

メイン手順との違い

事前準備（Bootstrap の作成・TCLSH 準備）と、リセット後の手順（初期プロンプト対応、Bootstrap 再作成、Controller Mode 有効化、SD-WAN 正常性確認）は、 メイン手順とまったく同じである。異なるのはリセットコマンドと OS イメージの扱いのみである。

  • リセットコマンド: factory reset all（factory reset all secure の代わり）。
  • OS イメージ: bootflash 上に残るため、USB/TFTP による OS 復旧は不要。
  • リセット後の起動: config-register 値に従って起動する（常に ROMMON へフォールバックするわけではない）。

factory reset all の実行

SD-WAN の Controller Mode で動作している状態から、以下のコマンドで Factory Reset を直接実行する。 

factory reset all

確認プロンプトが複数回表示されるため、内容を確認して Enter を押す。

禁止事項: Factory Reset 完了まで 30分から3時間程度かかる場合がある。Factory Reset 作業と再起動が確実に完了するまでは、 電源を抜かず、強制再起動や操作割り込みを行わないこと。

オプション: ROMMON モードで起動した場合

補足: Factory Reset 後に ROMMON モードで立ち上がってきた場合は、以下のコマンドでルーターを起動し直す。 要否は筐体型番に応じて判断する。 
rommon > confreg 0x2102
rommon > boot
ISR1100-4G / 6G の注意: ISR1100-4G / 6G では、ROMMON 操作時にコンソールのボーレートを一時的に 115200 へ変更する必要がある場合がある。 起動後は元のボーレートへ戻すこと。

メイン手順へ続く

リセット完了後の手順は、メイン手順と同じである。 初期プロンプト対応、Bootstrap ファイルの再作成、Controller Mode の有効化、SD-WAN 正常性確認を実施する。

C-3. どのコマンドを使うか

factory reset all secure 完全消去したい場合に使用する。RMA 返送、廃棄、侵害を受けた機器、機密情報を確実に抹消したい場合向け。起動イメージも消えるため、事前に公式イメージを準備する。
factory reset all 標準のリセット。機器を再利用する、別サイトへ移設する、設定レジスタとローカル起動イメージは残したい場合向け。
factory reset keep-licensing-info リセットは行うが、ライセンス情報を維持したい場合に使用する。再登録や SLAC 取得などの手間を避けたい場合向け。
factory reset sed SED ドライブのみ消去したい場合に使用する。C8550-G2 / C8570-G2 で、ブートフラッシュが SED 有効の場合が対象。ハードディスクは該当しない。

C-4. 実行前の共通注意

必須: 実行前にイメージ、設定、個人データを必ずバックアップする。 リセット中は電源を切らない。ISSU / ISSD が進行中の場合は実行しない。 適用済みソフトウェアパッチはリセット後に復元されない。 ZTP が設定済みの場合、リセット完了後に ZTP 構成で再起動する。
